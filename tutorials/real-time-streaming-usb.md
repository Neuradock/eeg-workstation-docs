# USB Real-Time Data Streaming

This tutorial explains how to read real-time EEG data from **NeuraDock EEG Workstation** through the USB workflow.


## Overview

NeuraDock EEG Workstation supports real-time EEG data streaming through USB.

Compared with the Bluetooth real-time streaming workflow, the USB real-time data structure is simpler:

- In the Bluetooth workflow, one row may contain 5 data blocks.
- In the USB workflow, one row contains only 1 data block.

Other parts of the workflow are generally consistent with the Bluetooth version.

Because the USB workflow sends one data block per row, the parser does not need to unpack 5 sampling blocks from each row. This makes the USB workflow easier to read and process in real time.

## Real-Time Streaming Concept

The real-time data stream can be understood as a continuous flow of EEG data from NeuraDock to the host computer.

The host program receives incoming data, groups it into manageable chunks, and then passes each chunk to the analysis or visualization workflow.

A typical USB real-time workflow is:

1. Connect NeuraDock EEG Workstation through USB.
2. Start the real-time data stream.
3. Receive data rows from the device or software interface.
4. Parse each row into EEG channel values.
5. Group the parsed samples into batches.
6. Send each batch to the downstream algorithm, visualization, or feedback workflow.

## USB Data Block Structure

In the USB API workflow:

```text
One row = one data block
```

This is different from the Bluetooth workflow, where one row may contain multiple data blocks.

The USB stream should be parsed into the default 7-channel layout:

```text
O1, O2, Oz, PO3, PO4, CP5, CP6
```

The recommended parsed output format is:

```text
sample_index, timestamp, O1, O2, Oz, PO3, PO4, CP5, CP6, marker
```

If timestamp or marker information is not available in the stream, those fields can be omitted or generated by the receiving program.

## Difference from Bluetooth Streaming

| Item | USB Real-Time Streaming | Bluetooth Real-Time Streaming |
|---|---|---|
| Row structure | One row contains one data block | One row may contain 5 data blocks |
| Parsing complexity | Simpler | Requires unpacking grouped blocks |
| Data grouping | Controlled mainly by the receiving program | May include packet-level grouping |
| Final parsed channel layout | O1, O2, Oz, PO3, PO4, CP5, CP6 | O1, O2, Oz, PO3, PO4, CP5, CP6 |
| Typical use case | Stable real-time data access through USB | Wireless real-time data access |

## `data_group_len`

The existing tutorial specifically notes that users should pay attention to the value of:

```text
data_group_len
```

`data_group_len` controls how many samples or data rows are grouped together before being passed to the downstream algorithm or visualization process.

This value affects the balance between:

- Algorithm performance
- Real-time responsiveness
- Processing delay
- Batch size
- System stability

## Choosing `data_group_len`

A smaller `data_group_len` may provide lower latency, because data is passed to the algorithm more frequently.

However, if the value is too small, the algorithm may not have enough data to process effectively, or the system may spend too much time handling many small batches.

A larger `data_group_len` may provide more stable algorithm input, but it may also increase delay.

Therefore, `data_group_len` should be selected based on the intended use case.

| Goal | Suggested Consideration |
|---|---|
| Lower latency | Use a smaller group length |
| More stable batch input | Use a larger group length |
| Real-time feedback | Balance latency and algorithm requirements |
| Offline-style processing | Larger groups may be acceptable |
| Real-time visualization | Avoid groups that are too large |

The exact recommended value should be confirmed based on the algorithm and workflow being used.

- Interactive media systems
- BCI experiment development
- Real-time agent-assisted EEG workflows

## Notes for Developers

When implementing or modifying the USB real-time streaming reader, keep the following points in mind:

- In the USB workflow, one row contains one data block.
- The parser should preserve the default channel order.
- `data_group_len` should be chosen according to the algorithm and latency requirements.
- Real-time workflows should avoid unnecessary blocking operations.
- Raw streamed data and parsed data should be clearly separated.
- The output format should be consistent with the offline USB text file reader where possible.
- If markers are used, marker timing and alignment should be documented clearly.
- If timestamp information is generated by the host computer, this should be stated clearly.
